01 / Code Walkthrough

Read the Python file in the same order the robot runs.

If someone checks out this project, they should not have to reverse-engineer the file structure. This section explains what each part of `Obj_track_adv.py` is responsible for and why it exists.

Code Map
1. `NETWORK_CONFIG`, `VISION_CONFIG`, `CONTROL_CONFIG`

These blocks are the setup layer. They answer practical questions: which phone camera stream to use, which YOLO model to load, what object to track, what follow distance looks like, and how aggressive the controller should be.

2. Helper classes

PID computes corrections from error, MobileVideoStream keeps the newest camera frame available, Commander sends wheel commands, and Telemetry listens for encoder feedback.

3. `Tracker`

This is the brain of the script. It loads YOLO, finds the target object, keeps track of the robot state, smooths noisy measurements, runs both PID loops, and mixes everything into left and right wheel speeds.

4. `main()`

The main loop wires all the pieces together. It starts the stream, creates the tracker, processes frames forever, sends commands on a fixed interval, draws the debug overlay, and shuts everything down safely at the end.

How To Think About The File
Input camera frame
Detection bounding box
Errors angle + area
Controller turn + distance PID
Output wheel speeds
Transport UDP command
The important design choice is that vision and motor driving are separated by a clean control layer.
The code does not say “ball is on the right, spin right.” It computes continuous error values and lets the controller decide how strongly to react.
That is why the robot can be tuned instead of hard-coded.

This block is not algorithmic. It just tells the script where the phone stream is and where to send motor and telemetry packets.

02 / Architecture

The full stack from camera frame to wheel torque.

The project is easiest to understand as a chain of responsibility. Each stage does one thing well and hands the next stage a simpler problem.

Phone camera stream Input stage

The phone is effectively a low-cost wireless vision sensor. It avoids running inference on-board while still delivering a real camera viewpoint from the robot's height.

`requests.get(..., stream=True)` reads MJPEG chunks and extracts JPEG frame boundaries.
The latest valid frame replaces the previous one inside a locked buffer so control code always sees the newest image.
Reconnect backoff handles temporary WiFi glitches without crashing the process.
What the helper classes are really doing
class PID:                # convert error into correction
class MobileVideoStream:  # keep the newest camera frame ready
class Commander:          # send UDP motor commands
class Telemetry:          # receive encoder feedback
class Tracker:            # combine vision + control
`PID.update(error)`

This method computes proportional, integral, and derivative terms from the current error. In this project, the turn PID and distance PID both reuse the same class, which keeps the control logic consistent.

`MobileVideoStream`

The camera reader runs in its own thread because waiting for network video inside the control loop would make the robot sluggish and unpredictable.

`Commander` and `Telemetry`

These two classes isolate networking details. That way, `Tracker` only worries about behavior and not about socket packet formatting.

03 / Hardware

The robot is simple on purpose.

The hardware choices keep the robotics interesting without making the wiring or embedded code the bottleneck. The heavy math lives on the laptop; the robot mostly needs clean actuation and stable telemetry.

Component Part Role
Microcontroller ESP32 WROVER Receives UDP commands, drives PWM channels, and returns encoder telemetry without carrying the ML workload.
Motor driver TB6612FNG Converts logic-level direction and PWM signals into bidirectional motor drive for the left and right wheels.
Drive motors DG01D gear motors Differential drive lets steering emerge from left/right speed differences instead of requiring a servo steering rack.
Vision sensor Smartphone stream Acts as a movable WiFi camera and gives the model the exact perspective the robot sees in deployment.
Compute Laptop or desktop Runs YOLO, smoothing, PID control, and visualization. CUDA is used when available and CPU remains a fallback.
Motor sign convention

The command path sends -left_speed, right_speed because one motor is mounted in reverse. That inversion is not a hack in the UI; it reflects the actual drivetrain orientation.

Power note

The motors should not share a noisy, under-regulated rail with the ESP32. Brownouts and encoder glitches are often power problems, not software problems.

Telemetry note

Encoder ticks are already flowing back to Python. That is enough structure to add closed-loop wheel-speed regulation later without changing the network protocol.

04 / Vision + Control

The updated Python loop is more honest about what the robot is doing.

The page now mirrors the code more closely. The tracker smooths horizontal error, uses the distance PID output for forward and reverse motion, scales motion by state, and keeps the tunable values in one obvious config section near the top of the Python file.

Angle error

(cx - width / 2) / width measures horizontal offset from the image center. Negative means the ball is left of center, so the robot should steer left. Positive means the opposite.

Distance error

(area - target_area) / target_area compares the current apparent object size to the desired size. If the ball looks too large, the robot is too close and should back up.

Why normalized

These values are normalized instead of raw pixels, which makes the controller easier to understand and less tied to one camera resolution.

How the script computes the two main errors
cx = (x1 + x2) / 2
area = (x2 - x1) * (y2 - y1)

angle_error = (cx - width / 2) / width
dist_error  = (area - target_area) / target_area
Center position math

The box coordinates are x1, y1, x2, y2. Averaging x1 and x2 gives the horizontal center of the detected object, which tells the robot whether the target is left or right.

Steering error math

The term cx - width / 2 measures how far the object is from the center line of the image. Dividing by width turns that into a normalized fraction instead of a raw pixel value.

Distance proxy math

The robot does not know real distance directly from one image, so it uses bounding-box area as a proxy. Bigger box means the object is closer; smaller box means it is farther away.

Low-pass smoothing

smoothed_angle += alpha * (raw - smoothed) means the controller does not trust one frame completely. It moves partway toward the latest measurement, which removes jitter.

Distance PID

The distance PID decides whether the robot should move forward or backward. A positive distance error means the ball is too close, so the script makes base_speed negative and backs away.

State scaling

ACQUIRING limits speed while detections stabilize, SEARCHING rotates using the last known direction, and STOPPED resets controller memory so old errors do not leak into the next run.

How errors become wheel commands
smoothed_angle += alpha * (raw_angle_error - smoothed_angle)
turn_cmd = turn_pid.update(smoothed_angle)
dist_cmd = dist_pid.update(dist_error)

base_speed = -dist_cmd
tgt_left   = base_speed + turn_cmd
tgt_right  = base_speed - turn_cmd
Smoothing

smoothed_angle += alpha * (raw - smoothed) is a low-pass filter. It means the new angle estimate only moves partway toward the latest measurement, which reduces jitter from noisy detections.

Distance sign

base_speed = -dist_cmd matters because a positive distance error means the target looks too large, so the robot is too close. The minus sign turns that into a backward motion command.

Differential steering

Adding turn_cmd to the left wheel and subtracting it from the right wheel makes the robot rotate. If turn_cmd is positive, the left side runs faster and the robot turns right.

Top-of-file config

The top config blocks make the script approachable because most users can get the project running without editing the logic itself.

Safer shutdown

The try/finally structure is there so motors stop and sockets close even if the user quits unexpectedly. This is more than neatness; it is a safety feature.

Cleaner PID behavior

The integral term is clamped so that if `Ki` is enabled later, long periods of accumulated error do not explode into a huge delayed correction.

Updated control logic in `Obj_track_adv.py`
raw_angle_error = (cx - width / 2) / width
smoothed_angle += smooth_rate * (raw_angle_error - smoothed_angle)

dist_error = (area - target_area) / target_area
turn_cmd = turn_pid.update(smoothed_angle)
dist_cmd = dist_pid.update(dist_error)

# Positive distance error means the target is too close.
base_speed = -dist_cmd

tgt_left = base_speed + turn_cmd
tgt_right = base_speed - turn_cmd

tgt_left *= speed_limit
tgt_right *= speed_limit
Step 1: Measure

The code starts by turning the detection box into measurable quantities: center position and area. Without those, there is nothing to control.

Step 2: Smooth

The raw angle can jump frame to frame, so smoothing produces a more trustworthy steering signal.

Step 3: Correct

Turn PID handles left-right alignment, distance PID handles forward-backward spacing, and their outputs are mixed into wheel speeds.

In this project, a PID output is just a correction value. The turn PID output says how strongly the robot should rotate left or right. The distance PID output says how strongly the robot should move forward or backward.
Horizontal position and apparent object size describe different physical problems. One controls steering, the other controls spacing. Keeping them separate makes the math easier to tune and understand.

Robot behavior as a state machine

Stopped

No recent detection. Motors off, controller state reset, waiting for a valid target.

Searching

The ball was just lost, so the robot rotates in the last known direction instead of freezing immediately.

Acquiring

The ball reappeared. Speed is intentionally limited while the detection is confirmed over several frames.

Tracking

Full control is active. Turn and distance corrections are mixed into left/right wheel commands in real time.

05 / Firmware

The ESP32 stays lean so the robot stays responsive.

The firmware does not need to know about detection classes, confidence thresholds, or PID tuning. It only needs to convert simple packets into motor actions and send lightweight telemetry back.

Command packet

MOTOR,L,R,LIGHT keeps the protocol plain-text and easy to inspect while debugging over UDP.

Stop packet

STOP,OFF is sent at startup and shutdown so the robot fails safe even when the vision side restarts.

Telemetry packet

ENC,leftTicks,rightTicks,cmdLeft,cmdRight gives Python enough signal to visualize and later close the speed loop.

How the Python script sends a command
message = f"MOTOR,{int(left)},{int(right)},{light.upper()}"
sock.sendto(message.encode(), (esp_ip, udp_cmd_port))
Packet format

The command is plain text, which makes it easy to inspect while debugging. The ESP32 only has to split the string and interpret left speed, right speed, and the light state.

Why convert to integers

The math produces floating-point values, but motor drivers use discrete speed levels. Converting to int makes the command explicit before sending it over the network.

Why keep sending commands

The script sends commands repeatedly at a short interval. That means the robot stays responsive and a single dropped UDP packet usually does not change behavior much.

A stale motor command is worse than a dropped motor command. UDP lets the latest packet win without retransmission delays, which is exactly what a fast control loop wants.
Verify motor polarity and power first. Then tune turn responsiveness, then distance behavior, and only then consider enabling integral compensation.
Add wheel-speed PID using the existing encoder telemetry, or move inference onto an onboard SBC like a Jetson Nano or Raspberry Pi 5 to remove WiFi latency.
